# ohr - Project Instructions

> **apfel is the golden example.** All patterns, conventions, and guardrails from [apfel](https://github.com/Arthur-Ficial/apfel) apply here. When in doubt, check how apfel does it.

## The Golden Goal

ohr has ONE purpose with TWO delivery modes:

> **Expose Apple's on-device SpeechAnalyzer as a usable, powerful UNIX tool
> and an OpenAI API-compatible transcription server.**

### The two modes, in priority order:

1. **UNIX tool** (`ohr meeting.m4a`, `ohr --listen`, `ohr -o srt audio.wav`)
   - File in, text out. That simple.
   - Pipe-friendly: `ohr meeting.m4a | apfel "summarize this"`
   - Output formats: plain text, SRT subtitles, JSON with timestamps
   - Live microphone input with `--listen`
   - Works with `jq`, `xargs`, shell scripts
   - `--json` output for machine consumption
   - Respects `NO_COLOR`, `--quiet`, stdin detection
   - Correct exit codes

2. **OpenAI-compatible HTTP server** (`ohr --serve`)
   - Drop-in replacement for `POST /v1/audio/transcriptions`
   - Accepts audio files (m4a, wav, mp3, etc.)
   - Returns transcription in OpenAI response format
   - Same security model as apfel (origin check, token auth, CORS)
   - Honest 501s for unsupported features

### Non-negotiable principles:

- **100% on-device.** No cloud, no API keys, no network for inference. Ever.
- **Honest about limitations.** Say what formats work, what languages are supported, what quality to expect.
- **Clean code, clean logic.** No hacks. Proper error types.
- **Swift 6 strict concurrency.** No data races.
- **Zero dependencies.** Model ships with macOS 26. No downloads, no pip, no torch, no ffmpeg.

## Architecture

```
CLI (file/mic/stdin) ──┐
                       ├--> Transcriber.swift --> SpeechAnalyzer (on-device)
HTTP Server (/v1/*) ---┘    SpeechTranscriber --> real-time streaming
```

- `OhrCore` library: pure Swift, no Speech framework dependency, unit-testable
- Main target: SpeechAnalyzer/SpeechTranscriber integration, Hummingbird HTTP server
- Apple framework: `import Speech` (SpeechAnalyzer, SpeechTranscriber)
- Tests: `swift run ohr-tests` (pure Swift runner, no XCTest needed - same pattern as apfel)
- No Xcode required - builds with Command Line Tools only
- Requires macOS 26+, Apple Silicon

## Build & Test

```bash
make install                   # bump patch + build release + install to /usr/local/bin
make build                     # bump patch + build release
swift build                    # debug build
swift run ohr-tests            # run pure Swift unit tests
```

**Version is in `.version` file** (single source of truth). Same auto-bump pattern as apfel.

**Always use `make install` for testing changes.**

Integration tests (requires server running):
```bash
python3 -m pytest Tests/integration/ -v
```

## Handling GitHub Issues

Same process as apfel:

1. **Fetch** the full issue with `gh issue view <n> --repo Arthur-Ficial/ohr --json body,comments,title,author,labels`
2. **Vet** - does it align with the golden goal? Can you reproduce it?
3. **Fix** if valid - TDD, keep changes minimal, run all tests
4. **Release** if code changed - `make install`, package asset, `gh release create`, update homebrew tap
5. **Close** with a friendly, short, truthful comment
